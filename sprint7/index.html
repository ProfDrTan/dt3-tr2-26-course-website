What Sprint 7 Is Really About

Six sprints in, every team has a working pipeline of some shape. The gap between teams now isn't "does it run" — it's whether the history of how it was built is something a stranger (or a grader, or next trimester's cohort) could follow. Sprint 6 was about extending capability. Sprint 7 is about discipline: closing the loop on work that's been left open, and making sure "done" actually means done.

Sprint 7 goal in one sentence (draft — confirm wording)

By Sunday 19 July, every team's main branch reflects everything the team actually built this trimester — no abandoned branches holding real work, no open PRs nobody's looked at in two weeks, no automation that quietly overwrites what came before it.

Three New Requirements in Sprint 7 — PROPOSED, not confirmed
1 · Branch and PR cleanup — every team, no exceptions

Every currently-open PR must be either merged or explicitly closed with a one-line reason by the Sprint 7 deadline. Any branch containing real, working automation that hasn't reached main must be merged in. This directly follows from what showed up in two different teams' repos this week: real, working automation sitting on an unmerged branch, invisible to anyone not specifically checking — including, in one case, the team's own Discord claim running ahead of what was actually on main.

2 · Automation must never overwrite without a trace

Any script that writes output data must either version its output (dated or numbered files) or write to a path that's exclusively its own. No team's automation should be able to silently overwrite another run's data — manual or automated — with no warning and no way to recover it. (Recovery via git log on the affected file is always possible if the data was ever committed — this requirement is about not needing that safety net in the first place.)

3 · Calibration continues — cumulative W3–W7

R10 extends the cumulative accuracy table through Sprint 7. This is unchanged from Sprint 6's intent — the value of calibration compounds the longer it runs, and Sprint 7 is not the sprint to let that streak break.

Scrum Health This Sprint

Sprint 7's Scrum focus is the retrospective, done honestly. A team that can point at its own abandoned branches and explain why they happened — duplicate work, unclear ownership, a dead-end experiment — is demonstrating more maturity than a team with a spotless-looking history that never says anything happened at all.

Definition of Done — Write it down

Weak DoD: "cleaned up some branches." Strong DoD: every branch older than 2 weeks either merged or deleted with a reason logged · every open PR has a named owner and a merge-or-close decision · vW29 tag cut before Sunday midnight. Scrum principle: a backlog you never groom stops being a backlog and starts being a graveyard.

Process Notes
On why this sprint exists

Two separate incidents this week — one team with real, working automation stuck on an unmerged branch, another with a manual run silently overwriting a week of automated output — both trace back to the same root cause: branches and PRs that don't get closed out. Sprint 7 exists to fix that directly, for every team, before it costs someone their own data.

On "done"

Work that exists only on a branch nobody's looking at might as well not exist. Done means merged to main — not "the code technically runs somewhere."
